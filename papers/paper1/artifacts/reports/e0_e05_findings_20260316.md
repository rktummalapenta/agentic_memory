# Experiment 0 and 0.5 Findings Report

Date: 2026-03-16

## Scope

This report consolidates the current evidence from:

- E0 pilot, `tiers 2 3`, `30` sessions
- E0 pilot, `tiers 2 3`, `60` sessions
- E0 exploratory, `tiers 1 2 3`, `36` sessions
- E0 confirmatory, `tiers 1 2 3`, `60` sessions
- E0.5 exploratory reduced sweep, `36` sessions
- E0.5 confirmatory reduced sweep, `60` sessions
- E0/E0.5 final reduced confirmatory set, `120` sessions

The `120`-session reduced set is now the main result bundle for Paper 1.

## Final Concrete Findings

1. Memory benefit is no longer a pilot-only result. It is stable across every E0 run.
2. The main benefit appears in multi-turn work, not in simple controls.
3. `B wm2` is the current best overall and best efficiency configuration.
4. `C ep3` is competitive, but it is not beating `B wm2` often enough to justify the extra memory cost.
5. `D sem1` is useful in some domains, especially `sec_edgar`, but it is not the overall winner.
6. The current evidence supports selective memory sizing, not maximal memory sizing.

## Failure Pattern Summary

- `A` still collapses mainly through semantic mismatches and context loss.
- The largest remaining error class in all memory conditions is still `other_semantic_mismatch`.
- Memory removes a large amount of context failure, but it does not eliminate semantic/result-shape errors.
- This supports a precise paper claim: memory fixes continuity and follow-up reasoning more than it fixes all SQL-generation errors.

## Final Paper-Level Claims Supported Now

The data now supports these claims:

1. Memory materially improves multi-turn enterprise Text-to-SQL accuracy.
2. The benefit is concentrated in multi-turn complexity rather than single-turn controls.
3. Working memory provides most of the benefit.
4. Selective episodic retrieval can help, but high retrieval depth is not justified.
5. Narrow semantic retrieval can help in some domains, but it is not the general best default.
6. More memory is not monotonically better.
7. The current best operating point is small working memory with optional selective retrieval.

The data does not support these claims:

- one universal winning memory stack across all datasets
- large retrieval depth as a default strategy
- semantic memory as the dominant general-purpose layer

## Final Recommendation For Paper 1

Freeze the current Paper 1 narrative around:

- `E0`: memory benefit exists and is large
- `E0`: the benefit is mostly a multi-turn phenomenon
- `E0.5`: the efficient operating point is small working memory plus selective retrieval
- `E0.5`: memory should be sized carefully rather than maximized by default

The next missing work is statistical testing and confidence intervals, not more search over memory settings.
